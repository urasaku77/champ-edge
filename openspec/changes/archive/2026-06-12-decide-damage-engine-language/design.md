# デザイン: ダメージエンジンの実装言語

## 方針

- Phase 1 のダメージエンジンは **Dart での実装**を優先する。
- Flutter の shared codebase 内に収め、UI とロジックの連携をシンプルにする。
- 将来的に必要なら `Rust` などで再実装し、Dart から FFI で呼び出せる形を検討する。

## アーキテクチャ概要

1. Dart 共有ロジック層
   - ダメージ計算エンジンを Dart クラスとして実装
   - 入力モデル、計算パラメータ、結果モデルを明確に定義
   - UI 側から直接呼び出せる API を提供

2. 旧 Python ロジックの参照と移植
   - 旧リポジトリの `~/Documents/champ-edge/pokedata/calc.py` を基本参照とする
   - 既存ロジックのアルゴリズムをそのまま Dart に置き換える形で移植
   - 移植時に再利用可能な関数単位で整理し、テストしやすい構造にする

3. 性能とテスト
   - まずは正しい結果を返すことを優先し、動作確認後に計算パスを最適化
   - パフォーマンス測定を追加し、ログやベンチマークで 100ms 未満を確認
   - 計算ロジックは単体テストで網羅し、将来的な言語移行時にも比較可能にする

4. 将来の共有化への準備
   - Dart での実装が十分速くなければ、`Rust` 共有ライブラリ化の検討を早期に開始
   - その場合、現在の Dart 実装を仕様ドキュメントとして利用できるように整理

## トレードオフ

### Dart 直実装

- メリット
  - Flutter への統合が最もシンプル
  - ツールチェーンが統一され、開発速度が高い
  - テストやデバッグが Dart エコシステム内で完結

- デメリット
  - Python の既存ロジックを手動移植する必要がある
  - 将来的な他言語共通化には再実装が必要になる

### Rust 共有ライブラリ化

- メリット
  - 高性能な計算エンジンを共通化できる
  - 将来的な Web / デスクトップ展開にも再利用しやすい

- デメリット
  - FFI 経由での呼び出し実装とビルド管理の工数が増える
  - Phase 1 のリリーススケジュールではリスクが高い

### Dart + ネイティブモジュール併用

- メリット
  - 言語ごとの最適実装が可能

- デメリット
  - 実装が複雑になり、Phase 1 の迅速開発に不向き

## 実装上の注意

- まずは Dart で 1 つの計算パスを正しく動作させる
- その上で、性能が不足する場合は計算ロジックの最適化を行う
- 既存 `calc.py` のロジックと結果を比較できるテストを用意する
- `pokemon.db` 参照は別層に分け、計算ロジックそのものは純粋な関数として扱う
