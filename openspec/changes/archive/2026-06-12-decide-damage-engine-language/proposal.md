# ダメージエンジン実装言語の選定

## 概要

Phase 1 のダメージ計算エンジンをどの言語で実装し、どのようにモバイルアプリに共有するかを決定します。

## 推奨

このフェーズではまず **Dart での直実装**を推奨します。

## 目的

- Phase 1 MVP の迅速な開発
- Flutter の shared codebase との親和性
- オフラインで高速に動作するダメージ計算の実装

## 比較候補

| 案 | メリット | デメリット |
|---|---|---|
| **Dart 直実装** | - Flutter 環境でそのまま動く
- 言語切り替えが不要で実装コストが低い
- Flutter の UI とロジックを同一コードベースで管理できる | - Python の既存ロジックを移植するコストがある
- 将来の他プラットフォーム共有性は制限される
| **Rust で共通ライブラリ化** | - 高性能かつクロスプラットフォームで再利用可能
- 将来の Web / デスクトップ展開にも活用しやすい | - Rust 環境の学習コスト
- Flutter との FFI 連携の実装負荷
| **Dart + 共有ネイティブモジュール** | - 言語ごとの最適実装が可能
- 既存の Python ロジックを部分的に再利用しやすい | - 実装構成が複雑になり、Phase 1 の開発速度が落ちる
- ネイティブバイナリの管理やテストが増える

## 推奨理由

- Phase 1 では開発リソースが限られているため、最も実装コストが低い方法を優先する。
- Flutter を選定済みのため、Dart で実装すれば UI とビジネスロジックを同じ言語で管理できる。
- まずは Dart で動作する実装を作り、必要に応じて将来の `Rust` 共有ライブラリ化を再検討する。

## この変更で決める範囲

- ダメージ計算エンジンの Phase 1 実装言語
- Flutter 内での共有コードとしての扱い方
- 将来の共有ライブラリ化を見据えた移植方針

## これを決めない範囲

- 具体的なダメージ計算ロジックの詳細（旧 `calc.py` からの移植は別途実装）
- OCR やスクリーンショットの実装方式
- バックエンド側の同期方式

## 判断基準

- Phase 1 のダメージ計算が **100ms 未満**で応答できる設計である
- Flutter の shared codebase に自然に統合できる
- 移植作業が Phase 1 のリリーススケジュールに収まる
- 将来の拡張で Rust 共有ライブラリ化や別プラットフォーム展開を検討できる余地を残す
