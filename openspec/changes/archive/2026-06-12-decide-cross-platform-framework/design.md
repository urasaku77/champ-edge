# デザイン: クロスプラットフォームフレームワーク選定

## 方針

- Phase 1 のモバイルアプリには **Flutter** を採用します。
- 共有の Dart UI とビジネスロジック層を持つ Flutter プロジェクトとして構築します。
- 必要に応じて iOS/Android のネイティブ統合は **プラットフォームチャネル** で実装します。

## アーキテクチャ概要

1. 共有アプリ層
   - Flutter / Dart による UI と画面遷移
   - パーティデータ、ダメージ計算入力、オフライン同期状態の共有状態管理
   - 参照データやキャッシュデータのローカルアクセス層

2. ローカルストレージ
   - `drift` や `sqflite` など Flutter に適した SQLite ソリューションを検討
   - `assets/data/pokemon.db` を参照データとして読み込み、Dart から利用しやすいキャッシュに変換
   - Phase 1 MVP の全ての参照検索でオフライン読み取りを保証

3. ダメージ計算
   - `~/Documents/champ-edge/pokedata/calc.py` の計算ロジックを参考に、Dart でコア計算エンジンを実装
   - **100ms 未満の更新**要件を満たすために、計算パスを効率化
   - 必要に応じて将来の共有ライブラリ化を見据えた構造にする

4. OCR とスクリーンショット処理
   - Flutter UI とネイティブの iOS/Android スクリーンショット・OCR API をプラットフォームチャネルで接続
   - クロスプラットフォームのアプリシェルは Flutter で保持し、プラットフォーム固有機能は薄いネイティブアダプタで分離
   - OCR エンジンの具体的な選定は別の意思決定に委ねる

5. アプリのライフサイクルと画面向き
   - iOS、Android 両方でデフォルトを **ランドスケープ** に設定
   - ランドスケープを主軸とし、必要に応じて一部フローでのみ縦向き対応を検討

## フレームワークのトレードオフ

### Flutter

メリット
- 横画面やカスタム UI の表現力が高い
- インタラクティブなアプリでも性能が出しやすい
- 1つのコードベースでメンテナンスしやすい
- モバイル向けパッケージエコシステムが成熟している

デメリット
- Dart に習熟が必要
- OCR やスクリーンショットなどネイティブ統合は設計に注意が必要

### React Native

メリット
- JavaScript/TypeScript の知見があれば導入が早い
- エコシステムとライブラリが豊富

デメリット
- 計算重視 UI ではパフォーマンス調整が必要になる可能性がある
- スクリーンショット/OCR 連携のブリッジ設計が複雑になりやすい
- 横画面前提の一貫した描画を両プラットフォームで保証するのがやや難しい

### Kotlin Multiplatform / Compose Multiplatform

メリット
- ビジネスロジックの共通化に強みがある
- Android ではネイティブ性能が高い

デメリット
- iOS 側 UI の成熟度が低く、プラットフォーム固有実装が多くなる
- Phase 1 MVP の早期開発ではリスクが高い
- 横画面ゲーム補助アプリとしてのクロスプラットフォーム対応は事例が少ない

## 実装上の注意点

- Flutter コードベースはモジュール化し、プラットフォーム固有のフローを分離する
- ドメインデータのソースとして `assets/data/pokemon.db` を活用し、Dart から扱いやすいデータアクセス層を選ぶ
- 早期に小さな Dart 版ダメージエンジンで性能プロトタイピングを行う
- フル UI を構築する前に、Flutter からスクリーンショットと OCR フローが問題なく扱えることを検証する
