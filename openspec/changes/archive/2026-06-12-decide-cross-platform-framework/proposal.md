# クロスプラットフォームフレームワーク選定

## 概要

champ-edge-mobile Phase 1 MVP のためのクロスプラットフォームモバイルフレームワークを選定します。
iOS / Android の横画面対応、ネイティブ連携、オフライン性能、保守性を重視して判断します。

## 推奨

Phase 1 では **Flutter** を主要なクロスプラットフォームフレームワークとして採用します。

## 技術候補の比較

| 案 | メリット | デメリット | 流行・評価 |
|---|---|---|---|
| **Flutter** | - iOS / Android を単一のコードベースで実装できる
- 横画面 UI の制御性が高い
- 描画性能が良く、安定した高速表示が期待できる
- OCR や画像アクセス、ローカルストレージ関連のパッケージが揃っている | - Dart 言語の習得が必要
- ネイティブ機能はプラグインやプラットフォームチャネル設計が必要 | モバイルクロスプラットフォームの中で現在最も勢いがあり、UI重視案件で採用が急増中。
| **React Native** | - JavaScript / TypeScript で開発できる
- Web / React 経験を活かせる
- ライブラリが豊富で導入を始めやすい | - ネイティブブリッジ経由の連携が複雑になりやすい
- 高頻度更新やカスタムUIではパフォーマンス調整が必要 | 依然広く使われているが、Flutter に比べると近年の新規案件での成長は鈍化気味。
| **Kotlin Multiplatform** | - ビジネスロジックを Kotlin で共通化できる
- Android ネイティブ性能が高い | - iOS 側の UI は別実装となり、完全共通 UI は難しい
- 実務事例はまだ限定的で学習コストが高い | ビジネスロジック共通化の注目度は上がっているが、UI共通化の面では主流ではない。

## Flutter を推奨する理由

- iOS / Android 両対応を一つのコードベースで進めやすい
- 横画面中心の UI 設計、カスタム描画、パフォーマンス制御に強い
- 目標とする **100ms 未満でのダメージ計算結果表示**に適した安定したフレームワーク
- `assets/data/pokemon.db` などのローカルデータアクセスを Flutter から扱いやすい
- 初期 MVP の開発効率が高く、段階的な機能追加も管理しやすい

## この変更で決める範囲

- モバイルアプリの基本フレームワーク選定
- 共有 UI / 共有ロジック層のアーキテクチャ方針
- iOS/Android で共通化しつつ、必要なネイティブ統合ポイントを設計する方針

## この変更で決めない範囲

- ダメージ計算エンジンの実装言語や共有方式
  - 別途 `/opsx:propose decide-damage-engine-language`
- OCR 実装方式（OSネイティブ OCR / クラウド OCR など）
  - 別途 `/opsx:propose decide-ocr-implementation`
- バックエンド全体アーキテクチャ（モバイル側同期要件以外）

## 判断基準

- Phase 1 MVP 範囲において、同じコードベースで **iOS と Android** の横画面対応が実現できること
- オフライン時に **ダメージ計算**、**パーティ閲覧**、**キャッシュ済み参照データ** が問題なく動作すること
- スクリーンショット自動取込や OCR 処理に対して、iOS/Android の **ネイティブフック** を組み込めること
- UI の計算結果表示が **100ms 未満**で更新できる見通しがあること
- 開発リソースが限られている中で、プラットフォームごとの UI 書き換えを最小化できること

## 期待される成果

- Phase 1 用の Flutter ベースのモバイルアプリ骨子
- UI とアプリロジックを共有する Dart コードベース
- スクリーンショット / OCR 用のネイティブプラグインチャネル設計
- 共有ロジックとプラットフォーム固有機能をきれいに分離した構成
