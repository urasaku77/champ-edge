# デザイン: 対戦画面 UI

## Context

旧 champ-edge は Tkinter デスクトップ版で、左=自分 / 右=相手の 2 カラムに
「パーティ＋選出」「選択ポケモンの基本情報」「技＋ダメージ」を並べ、天候・フィールド・
タイマー・カウンター・対戦記録などを周囲に配置していた。性格は 5×5 グリッドのポップアップ、
技/特性/持ち物は検索・候補選択、HP はバー表示だった。

モバイル版（Flutter, 横画面優先）では画面が狭いため、同じ情報密度を保ちつつ操作を
タップ主体に再設計する。ダメージ計算エンジンは Dart 実装済み（`DamageCalc.calculateDamage`）で、
状態異常・天候・フィールド・壁・持ち物・テラス・急所・じゅうでん等の補正に対応済み。
UI 側はこれらの「入力」を与えるだけでよい。

## Goals / Non-Goals

**Goals:**
- 旧アプリの操作体系（左右対戦・パーティ/選出・選択ポケモン編集・技ダメージ）を踏襲する。
- 計算に影響する入力（状態異常・じゅうでん・急所・天候・フィールド・壁・テラス）を UI から指定できる。
- 技は文字入力で検索して選べる。性格は表形式で選べる。HP はバーで視認できる。
- ステータス（努力値・個体値・レベル）を編集でき、実数値とダメージが即時再計算される。
- 横画面のスマホ画面に 2 カラムが収まり、各カラムはスクロールで全項目に到達できる。

**Non-Goals:**
- ダメージ計算ロジックの変更（実装済み・対象外）。
- スクリーンショット OCR による自動入力（別 change）。
- 対戦記録の永続化・統計（メニュー項目はプレースホルダから段階導入）。
- ダブルバトル専用 UI（エンジンは対応済みだが UI は Phase 後半）。

## Decisions

### レイアウト
- 全画面・タイトルバーなし。隅（右上）のメニューボタンから Drawer を開く。
- `Row`[ 左パネル | 区切り | 右パネル ]、各パネルは `SingleChildScrollView`。
- 各パネル構成（上から）: 見出し → パーティ(6 画像) → 選出 → 選択ポケモン基本情報 → 技4。

### ポケモン表示・選択
- パーティ/選出/詳細はテキストではなく**画像**（`assets/pokemon/<no>-<form>.png`）で表示。
- パーティのポケモンを**タップで選出**（トグル）し、同時に詳細表示対象（active）にする。

### 編集（タップ→入力）
- **技**: タップ→**文字入力検索**つきの技選択（`waza_data` から名前部分一致で絞り込み）。
  DB 連携前はサンプル技プールを同じ検索 UI で代替。
- **特性 / 持ち物**: タップ→候補一覧から選択（特性はそのポケモンの候補、持ち物は一覧/検索）。
- **性格**: タップ→**5×5 表形式**（行=上昇能力↑、列=下降能力↓、対角＝補正なし）から選択。
- **ステータス**: タップ→編集ダイアログ（レベル、努力値 0–252、個体値 0–31）。実数値を表示。

### 状態・場の入力
- 選択ポケモンに**状態異常**（なし/やけど/まひ/どく/もうどく/こおり/ねむり）と
  **じゅうでん**・**急所**のトグルを置く（攻撃側で意味を持つものは攻撃側に）。
- メニューに**天候・フィールド**、**壁**（リフレクター/ひかりのかべ/オーロラベール）、
  **テラスタイプ**変更を置く。
- これらは `CombatantState` / `MoveState` / `FieldState` に反映してダメージ再計算する。

### HP バー
- 選択ポケモンに HP バーを表示。各技のダメージ割合（min–max%）をバー上にオーバーレイし、
  確定数（確定n発/乱数）も併記する。

### データ
- モデル `BattlePokemon` は種族値＋個体値＋努力値＋レベル＋性格から実数値を算出（実装済み）。
- 技/特性/持ち物の候補は DB（`pokemon.db`）から取得するのが最終形。DB 連携が未完の間は
  `lib/src/data/` のサンプル/オプションデータで動作させ、連携後に差し替える。

## Risks / Trade-offs

- **DB 連携依存**: 技の文字検索・全特性/全持ち物は DB 前提。連携前はサンプルで代替するため、
  「検索結果が限定的」になる過渡期がある。→ 検索 UI は DB 接続時にそのまま実データへ切替可能に設計。
- **画面の狭さ**: 横画面スマホに情報が多い。→ フォントを抑え、各カラムをスクロール可能にする。
  情報過多なら詳細をタブ/折りたたみに分割する余地を残す。
- **実数値の式差異**: 標準式（努力値0–252）で算出。エンジン検証で使った旧 `calc.py` の
  0–32 努力値スケールとは別系統だが、エンジンは「実数値」を受け取るため計算結果は一貫する。
- **状態項目の網羅**: エンジンは多数の補正に対応済みだが、UI に全部を出すと煩雑。
  Phase 1 は代表的な項目（状態異常・じゅうでん・急所・天候・フィールド・壁・テラス）に絞る。
